\doxysubsubsection{\texorpdfstring{AWS\_INTERLOCKED\_INT}{AWS\_INTERLOCKED\_INT}}
{\footnotesize\ttfamily \#define AWS\+\_\+\+INTERLOCKED\+\_\+\+INT(\begin{DoxyParamCaption}\item[{}]{x }\end{DoxyParamCaption})~\+\_\+\+Interlocked\#\#x\#\#64}

Copyright Amazon.\+com, Inc. or its affiliates. All Rights Reserved. SPDX-\/\+License-\/\+Identifier\+: Apache-\/2.\+0. Some general notes\+:

On x86/x86\+\_\+64, by default, windows uses acquire/release semantics for volatile accesses; however, this is not the case on ARM, and on x86/x86\+\_\+64 it can be disabled using the /volatile\+:iso compile flag.

Interlocked$\ast$ functions implicitly have acq\+\_\+rel semantics; there are ones with weaker semantics as well, but because windows is generally used on x86, where there\textquotesingle{}s not a lot of performance difference between different ordering modes anyway, we just use the stronger forms for now. Further, on x86, they actually have seq\+\_\+cst semantics as they use locked instructions. It is unclear if Interlocked functions guarantee seq\+\_\+cst on non-\/x86 platforms.

Since all loads and stores are acq and/or rel already, we can do non-\/seq\+\_\+cst loads and stores as just volatile variable accesses, but add the appropriate barriers for good measure.

For seq\+\_\+cst accesses, we take advantage of the facts that (on x86)\+:
\begin{DoxyEnumerate}
\item Loads are not reordered with other loads
\item Stores are not reordered with other stores
\item Locked instructions (including swaps) have a total order
\item Non-\/locked accesses are not reordered with locked instructions
\end{DoxyEnumerate}

Therefore, if we ensure that all seq\+\_\+cst stores are locked, we can establish a total order on stores, and the intervening ordinary loads will not violate that total order. See \href{http://www.cs.cmu.edu/~410-f10/doc/Intel_Reordering_318147.pdf}{\texttt{ http\+://www.\+cs.\+cmu.\+edu/$\sim$410-\/f10/doc/\+Intel\+\_\+\+Reordering\+\_\+318147.\+pdf}} 2.\+7, which covers this use case.
